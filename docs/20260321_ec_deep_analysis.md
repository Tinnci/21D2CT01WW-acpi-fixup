# EC 固件深度分析报告

**生成日期**: 2026-03-21  
**平台**: ThinkPad Z13 Gen1 (21D2) / Z16 Gen1, NM-E161  
**EC SPI 芯片**: U8505 (Winbond W25Q256, 32MB)  

---

## 1. EC SPI Dump 结构总览

所有 EC SPI dump 均为 **33,554,432 bytes (32MB)**。EC 固件主体位于 SPI 低地址区域，绝大部分空间为空白 (0xFF 或 0x00)。

### 1.1 _EC 头部

所有 dump 在 **SPI 偏移 0x001000** 处有主 _EC 头部：
- Magic: `5F 45 43` (`_EC`)
- 版本字节: `01` (v1, FL2格式) 或 `02` (v2, 工程版格式)
- 后跟 total_size 和 data_size 字段 (little-endian uint32)

工程版 _EC 头部示例 (ver=2):
```
5f454302 40010500 00000500 01000000
         ^^^^^^^^ ^^^^^^^^
         total    data
```

发布版 FL2 _EC 头部示例 (ver=1):
```
5f454301 20010500 00000500 01000000
```

### 1.2 版权字符串

所有 dump 在 **偏移 0x002580** 包含:
```
(C) Copyright IBM Corp. 2001, 2005 All Rights Reserved
US Government Users Restricted Rights
```

### 1.3 SPI 布局分类

| 类型 | EC 固件区域大小 | 空白填充 | 代表文件 | 说明 |
|------|----------------|----------|---------|------|
| 早期工程 (rev0.2-0.3) | 208-232 KB | 99.3% FF/00 | EC0.07, EC0.08, EC0.26 | 最小化固件 |
| 标准工程 (rev0.4) | 256 KB | 99.2% FF | EC0.01, EC0.12 | 含 0x051000 处 4KB 附加块 |
| 自用机 (N3GHT15W) | 276 KB + 3MB尾部 | ~90% | N3GHT15W_z13_own | 实际使用中含 NVRAM/日志数据 |
| 量产版 (v1.02+) | 288 KB + 10MB+ | 66%-57% | N3GHT25W, N3GHT64W | 含双固件镜像 + 配置/日志区 |
| 异常 (EC0.11_rev0.4) | 32MB 多区域 | 4.2% | EC0.11_eng_rev0.4 | 非典型 dump，似乎包含完整 BIOS+EC 混合内容 |

## 2. EC SPI 数据区域详解

### 2.1 典型工程版 SPI 布局 (以 EC0.01 为例)

| 起始地址 | 结束地址 | 大小 | 内容 |
|----------|----------|------|------|
| 0x000000 | 0x001000 | 4 KB | Boot config / 引导块 |
| 0x001000 | 0x040000 | 252 KB | EC 固件主体 (_EC header + code) |
| 0x040000 | 0x051000 | 68 KB | 空白 (0xFF) |
| 0x051000 | 0x052000 | 4 KB | 附加配置块 |
| 0x052000 | 0x2000000 | ~31.7 MB | 空白 (0xFF) |

### 2.2 量产版双镜像布局 (以 N3GHT64W 为例)

N3GHT64W SPI dump 有 **2 个 _EC 头部**:
- 主镜像: **0x001000** (header: `5f45430240f1040000f0040001000000`)
- 备份镜像: **0x0A0000** (header 相同)

| 起始地址 | 结束地址 | 大小 | 内容 |
|----------|----------|------|------|
| 0x000000 | 0x04A000 | 296 KB | 主 EC 固件 (含引导块 + _EC) |
| 0x050000 | 0x051000 | 4 KB | 配置块 A |
| 0x0A0000 | 0x0E9000 | 292 KB | 备份 EC 固件 |
| 0x0EF000 | 0x0F0000 | 4 KB | 配置块 B (与 A 相同) |
| 0x100000+ | | ~10 MB | EC 运行时数据 / NVRAM / 日志 |

共 **2798 个 4KB 数据块 (10.9 MB)**，唯一块数 2718。

### 2.3 N3GHT25W (v1.02) — 过渡版

包含 2 个版本标识: **N3GHT22W** 和 **N3GHT25W**，表明此 SPI dump 保留了旧版 EC 镜像。

### 2.4 自用机 N3GHT15W

主固件区 0x000000-0x045000 (276KB) 加上:
- 0x1C50000-0x1CB7000 (412 KB) — **旧版/备份 EC 固件代码** (ARM Thumb-2)
- 0x1CB7000-0x1D00000 (292 KB) — 间隔空白 (87.7% FF)
- 0x1D00000-0x2000000 (3 MB) — **EC Flash 磨损均衡日志** (交错存储的当前 EC 代码页)

> **⚠️ 修正说明**: 这两个区域最初被标记为"运行时 NVRAM/日志"和"大块运行时数据"。经深度逆向分析 (见第 7 节)，确认两者均为 **ARM Thumb-2 可执行代码**，分别是旧版固件备份和 Flash 页写入日志。

### 2.5 异常文件: EC0.11_eng_rev0.4

此文件数据密度极高 (**7872 个非空块 = 30.8 MB**)，254 个 _EC magic 匹配（绝大多数是误匹配），版权字符串偏移在 0x165191 而非标准 0x002580。

**推测**: 此文件并非纯 EC SPI dump，可能是完整编程器 dump 包含 BIOS+EC 混合内容，或是错误标记的文件。

## 3. 独立 EC 固件对比 (firmware/ec/)

### 3.1 文件清单

| 文件名 | 大小 | EC 版本 | _EC ver | total_size | data_size |
|--------|------|---------|---------|------------|-----------|
| N3GHT64W_v1.64_standalone.bin | 327,680 | N3GHT64W | 02 (v2) | 0x0004F140 | 0x0004F000 |
| N3GHT68W.FL2 | 327,968 | N3GHT68W | 01 (v1) | 0x00050120 | 0x00050000 |
| N3GHT69W.FL2 | 327,968 | N3GHT69W | 01 (v1) | 0x00050120 | 0x00050000 |

### 3.2 格式变化

- **N3GHT64W**: ver_byte=**02** (256B 对齐头部)，total=0x4F140, data=0x4F000，大小 327,680
- **N3GHT68W/69W**: ver_byte=**01** (FL2 格式)，total=0x50120, data=0x50000，大小 327,968 (+288B)

头部版本从 02→01 的变化表明 Lenovo 在 v1.64 到 v1.68 之间**切换了 EC 固件打包格式**。

### 3.3 版本间差异

| 对比 | 字节差异 | 差异率 | 差异区间数 |
|------|---------|--------|-----------|
| N3GHT64W vs N3GHT68W | 265,281 | 81.0% | 293 |
| N3GHT64W vs N3GHT69W | 267,544 | 81.6% | 63 |
| N3GHT68W vs N3GHT69W | 249,127 | 76.0% | 523 |

所有版本间差异率都在 **76-82%**，说明 EC 固件代码在这些版本间有大量重构/重编译，而非简单的补丁修改。

### 3.4 差异特征

N3GHT64W 头部字节 [3] = `02` → N3GHT68W/69W 头部字节 [3] = `01`  
offset 0x14-0x17 变化 — 可能是构建时间戳/校验:
- N3GHT64W: `92 6D 81 69`
- N3GHT68W: `81 80 B0 39`
- N3GHT69W: `78 A4 B9 15`

主代码差异集中在 0x001568 之后，即 _EC 头部后约 5KB 处开始的实际固件代码区。

## 6. 关键结论

1. **EC 固件起始地址**: SPI 偏移 **0x001000**，前 4KB 为引导/配置区
2. **双镜像机制**: 量产版 SPI 在 0x001000 和 0x0A0000 各有一份 EC 固件副本
3. **格式变化**: v1.64 (N3GHT64W) 使用 ver_byte=02，v1.68+ 切换到 ver_byte=01 (FL2 格式)
4. **版本间差异巨大**: 76-82% 字节差异率表明大量代码重构，非增量补丁
5. **EC0.11_rev0.4 异常**: 30.8MB 数据密度不符合 EC SPI dump 模式，建议进一步调查
6. **自用机 N3GHT15W**: ~~运行时数据~~ → 实为旧版 EC 固件备份 (R1, 412KB) + Flash 磨损均衡日志 (R2, 3MB)，详见第 7 节

## 7. N3GHT15W SPI 尾部区域逆向分析

**分析对象**: SPI 偏移 0x1C50000 (R1, 412KB) 和 0x1D00000 (R2, 3MB)  
**分析日期**: 2026-03-21  
**主 EC 参考**: SPI[0x1000:0x45000] (272KB, version N3GHT15W)

### 7.1 ARM Thumb-2 代码确认

两个区域最初被归类为"NVRAM/日志" (R1) 和"运行时数据" (R2)。通过字节频率统计、函数序言 (prologue) 检测和指针分析，确认 **均为 ARM Thumb-2 可执行代码**：

| 指标 | R1 (412KB) | R2 (3MB) |
|------|-----------|----------|
| PUSH/POP 函数序言 | 258 个 | 1,607 个 |
| 代码指标得分 | 13,607 (6.5%) | 98,012 (6.2%) |
| SRAM 指针 (0x2000xxxx) | 492 | 5,084 |
| Flash 指针 (0x0000xxxx) | 407 | 3,059 |
| 有效字符串 | 299 (多为噪音) | 2,609 (多为噪音) |

最高频字节: 0x00, 0xFF, 0x20, 0xF7, 0xF8, 0xF0, 0x46 — 与 ARM Thumb-2 指令编码特征一致。

### 7.2 R2 (0x1D00000-0x2000000): Flash 页写入日志

#### 结构

R2 共 3,145,728 字节 = 768 个 4KB 页，组织为 **48 组 × 16 页**。**每一页与主 EC 固件精确匹配 4096/4096 字节 (100%)**，零误差、零部分匹配。

#### 映射公式

```
EC_offset = 0x1C000 + (page % 16) × 0x1000 + (group) × 0x100 + 1
```

其中: group ∈ [0, 47], page ∈ [0, 15]

覆盖 EC 范围: 0x1C000 ～ 0x2DF00 (约 72KB 的 EC 代码区，位于 EC 偏移 112KB-184KB)

#### +1 字节偏移

每个 R2 页从 EC 对齐点的 **第 2 个字节开始存储** (跳过第 1 字节)。被跳过字节无特殊分布模式 (0x00 出现 10 次, 0x04 出现 8 次, 0x10 出现 8 次等)，为普通 ARM 指令字节。

#### XOR 验证

R2 首 256 字节与 EC[0x1C001] 的 XOR 结果 = **全零** — 无加密、无混淆，明文完全一致。

#### 解读

R2 中 48 组数据覆盖相同的 16 个 EC 代码页 (64KB 范围)，每组之间 EC 偏移步进 256 字节。这是 **SPI Flash 磨损均衡 (wear-leveling)** 或 **固件更新日志 (update journal)** 的典型特征：EC 控制器在写入 Flash 页时不覆盖原位置，而是追加写入到 R2 区域，保留旧页状态以支持回滚。

### 7.3 R1 (0x1C50000-0x1CB7000): 旧版 EC 固件备份

#### 结构

| R1 偏移 | 大小 | 内容 |
|---------|------|------|
| 0x000-0x0FE | 255 B | ARM Code (函数序言/代码片段) |
| 0x0FF-0xFFFF | 63.8 KB | ≡ EC[0x1B600:0x2B501] **完全一致** |
| 0x10000-0x66FFF | 348 KB | 不同版本的 ARM 代码 |

#### 关键发现

- R1[0xFF:0x10000] 与主 EC[0x1B600:0x2B501] **逐字节完全匹配** (65,281 字节)
- 首个差异出现在 R1[0x10000]: R1=0x17, EC=0x00 — 恰好在 64KB 边界
- 匹配后的 348KB 不匹配任何已知 EC 版本 (N3GHT64W, N3GHT68W, N3GHT69W)
- R1 前 255 字节为合法 ARM Thumb-2 指令 (PUSH/POP/BL/MOV 等)

#### 解读

R1 是一个 **旧版/过渡版 EC 固件残留**。第一个 64KB 扇区已被更新到当前 N3GHT15W 版本 (与主 EC 64KB 范围完全一致)，但后续 348KB 仍保留先前版本的代码。此模式符合 **扇区级固件更新** 机制：EC 更新时按 64KB 扇区粒度逐步覆写，R1 记录了更新到 N3GHT15W 时的中间状态。

### 7.4 交叉 Dump 对比

这两个区域在其他 SPI dump 中均为 **全 FF (空白)**：

| Dump | R1 区域 | R2 区域 |
|------|---------|---------|
| N3GHT15W (自用机) | ✅ 有数据 (412KB) | ✅ 有数据 (3MB) |
| EC0.11_eng_rev0.4 | ✅ 有数据 | ✅ 有数据 |
| EC0.01, EC0.07, EC0.08 等 | ❌ 全 FF | ❌ 全 FF |
| N3GHT25W, N3GHT64W 等 | ❌ 全 FF | ❌ 全 FF |

仅在实际运行过的设备 (N3GHT15W) 和特殊工程 dump (EC0.11) 中存在数据，进一步支持这些区域为 **运行时产生的 Flash 管理数据** 而非出厂预置内容。

### 7.5 完整 SPI 布局 (修正版)

```
N3GHT15W 自用机 EC SPI Flash (32MB) 完整布局:

0x0000000-0x0001000  引导/配置区 (4KB)
0x0001000-0x0045000  主 EC 固件 N3GHT15W (272KB, _EC header + ARM Thumb-2 code)
0x0045000-0x1C50000  空白/系统保留 (~28.3MB, 0xFF)
0x1C50000-0x1CB7000  旧版 EC 固件残留 (412KB, 部分已更新)
0x1CB7000-0x1D00000  间隔空白 (292KB, 87.7% FF)
0x1D00000-0x2000000  Flash 页写入日志 (3MB, 768页 = 48组×16页, 交错映射)
```

## 8. EC 固件基地址与 Cortex-M 向量表

**分析日期**: 2026-03-21


### 8.4 启动流程

```
Reset:     0x100701F4  ← 向量表 → CRT0 初始化
  ↓
CRT0:      0x10070240  ← 初始化 BSS/数据段
  ↓

  ↓

  ↓
SP 加载:   SP = 0x200C8000 (SRAM 顶部)
  ↓
main:      0x10070141  ← 跳转到主程序入口
  ↓
EC main:   0x1007166C  ← 第一个 PUSH 函数 (系统初始化)
```

## 9. 跨版本分析与 LADD 描述符

### 9.1 函数签名匹配 (R1 残留代码 348KB)

使用 PUSH {.., LR} 序言提取 200 个函数签名 (首 8 字节)，与各版本匹配：

| 目标版本 | 匹配率 | 说明 |
|---------|--------|------|
| N3GHT15W (当前) | 52.5% (105/200) | 最高，最接近当前版本 |
| N3GHT25W | 41.0% (41/100) | 中度相似 |
| N3GHT64W | 20.0% (20/100) | 较低相似度 |
| EC0.11_eng | 20.0% (20/100) | 工程版，差异大 |

R1 残留代码在其他 dump 中未找到完整匹配 (仅 33-103 字节的片段)，进一步确认其为独立的旧版本固件残留。

### 9.2 N3GHT25W/64W 双固件结构

N3GHT25W 和 N3GHT64W 在 SPI 0x100000 处包含 **LADD 描述符表**，定义 Flash 区域布局：

```
LADD 描述符 (SPI 0x100000, 两版本完全一致):
  [ 0] addr=0x00E47000  size=0x00080000 (512KB)
  [ 7] addr=0x000C5000  size=0x00380000 (3.5MB)
  [ 8] addr=0x00445000  size=0x00051000 (332KB)
  [ 9] addr=0x0158F000  size=0x009B0000 (10MB)
  [13] addr=0x00000000  size=0x00100000 (1MB)
```

这些 dump 在 SPI 0x100000 以上包含大量 ARM 代码区域：

| Dump | 区域 A | 区域 B | 区域 C | 架构 |
|------|--------|--------|--------|------|
| N3GHT25W | ~150KB | ~720KB | ~890KB | Thumb-2 + SVC #0xF2 |
| N3GHT64W | ~150KB | ~720KB | ~890KB | ARM 32-bit (bootloader) |

N3GHT64W 区域 B 入口代码 (`ldr ip, [pc]; ldr sp, [ip]; blx ip`) 为标准 ARM 32-bit bootloader 模式。
N3GHT25W 区域 B 包含 `svc #0xf2` 系统调用，为 RTOS 层代码。

### 9.3 N3GHT25W 双版本镜像

N3GHT25W dump 包含双版本：
- **主版本**: N3GHT25W (primary)
- **备份版本**: N3GHT22W (backup)
- 字节匹配率仅 19.5% — 版本差异显著

### 9.4 EC0.11_eng_rev0.4 异常结构

- 21.9MB 连续 ARM 代码块 (0x092A000-0x1E0F000)
- 无 _EC 头部标记，无 N3GHT 版本字符串
- IBM 版权位于异常偏移 0x165195 (标准位于 0x002584)
- R2 区域有数据但交错映射匹配率为 0% (不同于 N3GHT15W 的 100%)

## 10. 跨EC固件基地址/向量表/启动流程对比

### 10.1 分析方法

使用 `scripts/ec_cross_base_analysis.py` 对全部 13 个 SPI dump + 3 个独立 EC 文件进行自动化扫描：
1. 定位所有 `_EC` header → 提取 EC blob
2. 对每个 blob 测试 21 个候选基地址 (0x10060000-0x10090000 步进 0x1000)
3. 统计 LDR T1 (Thumb 短 LDR) 的 flash self-ref 数量，最高者为最佳基地址
4. 在 EC+0x0100/0x0120 扫描 Cortex-M 向量表 (SP + Thumb function pointers)
5. 检查启动流程标志 (base_ref literal pool)

### 10.3 关键发现

#### 10.3.1 基地址：所有版本均为 0x10070000

初步扫描显示两个分组 (0x10060000 / 0x10070000)，但深入分析证实这是**假阳性**：

- **N3GHT14T/11M/12W**: 所有 flash 引用目标都在 `[0x10070000, 0x100B0000)`，恰好落在 0x10060000 和 0x10070000 两个窗口的**重叠区域**。扫描算法在并列时首选较低候选值，导致误报。
- **N3GHT15W → 69W**: 有额外 253 个引用指向 `[0x100B0000, 0x100C0000)` (blob 尾部 64KB)，此区域仅在 base=0x10070000 时落入窗口内，因此明确胜出。
- 早期版本 (N3GHT14T 等) 的固件较小，不使用最后的 64KB 段（0x100B0000+），所以无法区分。

**结论: 所有 EC 版本的基地址统一为 0x10070000。**

#### 10.3.2 向量表偏移与 _EC header 版本关联

- **_EC header ver=2**: 向量表在 EC+0x**0100** (header 256 字节)
  - Copyright 在 EC+0x01584
- **_EC header ver=1**: 向量表在 EC+0x**0120** (header 288 字节，多 32 字节)
  - Copyright 在 EC+0x015A4 (偏移多 0x20)
  - 仅见于 N3GHT68W/69W 的 .FL2 独立文件

#### 10.3.3 SP (栈指针) 演进

- **0x200C8000**: 早期至中期版本 (N3GHT05T → N3GHT25W)
  - SRAM 区域: 0x200C0000-0x200C8000 = 32KB 完整使用
- **0x200C7C00**: 较新版本 (N3GHT64W, 68W, 69W)
  - 比 0x200C8000 少 1KB (0x400 字节)
  - 可能用于 MPU 保护区、栈溢出守护页或硬件保留

#### 10.3.4 完全不变的常量 (所有版本)

- **Reset handler**: 0x100701F5 (无一例外)
- **NMI handler**: 0x10070289/0x10070291 (两种变体)
- **HardFault handler**: 0x10070295/0x1007029D (两种变体)
- **有效向量数**: 79 (16 个系统 + 63 个 IRQ)
- **Copyright 字符串**: 始终存在 "Copyright IBM" / "Copyright LENOVO"

#### 10.3.5 N3GHT25W/64W 大型区域

在 SPI 0x100000 以上发现额外 ARM 代码区域:
- SPI 0x100000-0x124000 (144KB): LADD 描述符表区域，base=0x00000000
- SPI 0x12A000-0x1D7000 (692KB): 未知 ARM 区域，base=0x00200000
- 更多区域分析因运行时间过长被中断，待后续深入

### 10.4 版本演进时间线

```
早期工程版 (224KB, base=0x10070000):
  N3GHT05T → N3GHT07T → N3GHT08M

量产版过渡 (320KB, base=0x10060000):
  N3GHT11M → N3GHT12W → N3GHT14T

量产版成熟 (316-320KB, base=0x10070000, SP=0x200C8000):
  N3GHT15W → N3GHT22W → N3GHT25W

最新版 (316-320KB, base=0x10070000, SP=0x200C7C00):
  N3GHT64W → N3GHT68W → N3GHT69W
```

### 10.5 EC0.11_eng_rev0.4 异常

此文件无 _EC header，未被自动分析。前次分析发现其包含 21.9MB 连续 ARM 代码块，结构与标准 EC firmware 完全不同，可能是完整 SPI flash 原始镜像或调试版本。
